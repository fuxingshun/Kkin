# 项目文件层级说明与清理建议

本文档按当前项目目录层级说明每个主要文件的作用，并标注是否可以删除。

## 状态说明

| 状态 | 含义 |
| --- | --- |
| 必留 | 当前小程序、后台管理端或 Java 后端运行需要 |
| 可选 | 不影响核心运行，但对文档、启动便利、开发体验有帮助 |
| 谨慎删除 | 可能包含本地配置、历史数据、上传文件或密钥，删除前要备份或确认 |
| 可删除 | 运行缓存、构建产物、旧链路残留或 IDE 本地文件，可以清理 |
| 可进一步精简 | 当前仍有业务功能引用，但如果产品范围继续缩小，可以按页面或功能一起删除 |

## 当前目标架构

当前项目建议保留四块：

| 模块 | 目录 | 说明 |
| --- | --- | --- |
| 微信小程序三端 | `miniapp/` | 老人端、家属端、服务人员端 |
| 后台监控管理端 | `src/admin/`、`apps/web/admin.html`、`config/vite.config.admin.ts` | Web 管理后台，目前以 React 单页实现 |
| Java 后端 | `server-java/` | Spring Boot、MySQL、MyBatis-Plus，负责 API、AI 语音、数据持久化 |
| 项目文档和脚本 | `docs/`、`scripts/`、根配置 | 说明、启动脚本、构建配置 |

旧的 Python 服务、Fay 数字人、H5 老人端、H5 家属端、MCP 服务已经不属于目标架构。


同时从根 `package.json` 删除：

```json
"start:all": "scripts/start-all.cmd",

```

## 后续结构优化建议

1. 后端继续拆分 mapper：把 `users`、`schedules`、`alerts`、`media`、`moods` 等表逐步改成 `Entity + BaseMapper + Service`。
2. 后台管理端从 `src/admin/main.tsx` 拆成多个组件和页面，并接真实 Java API。
3. 小程序 `app.css` 可以按老人、家属、服务人员拆分，清理不用的样式。
4. 如果产品继续瘦身，优先考虑删除“媒体回忆”和“咨询”相关页面，因为它们不是老人、家属、服务人员三端的最小闭环必需功能。
5. `server/` 目录可以逐步改名为 `runtime/` 或 `uploads/`，避免它继续看起来像旧 Python 服务端。
