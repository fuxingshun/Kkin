# 封闭试点备份恢复演练

本文档用于 KinEcho 封闭试点期间的数据备份和恢复演练。目标是在不影响线上家庭使用的前提下，定期确认数据库和上传文件可以被完整导出，并在隔离环境完成恢复验证。

## 备份范围

- MySQL 数据库：用户、家庭绑定、咨询、照护记录、预警、同意记录、隐私请求和审计日志。
- 上传目录：家庭媒体、缩略图、AI 语音回复和老人录音临时文件。
- 运行配置：生产环境变量、发布版本号和当日值班信息不写入备份包，需由运维台账单独保存。

## 执行备份

在已配置生产数据库环境变量的机器上执行：

```powershell
npm run backup:pilot
```

脚本会在 `.backups/kinecho-pilot-<timestamp>` 下生成：

- `manifest.json`：备份时间、数据库导出状态、上传目录归档状态和需人工补充的动作。
- `database.sql`：通过 `mysqldump` 导出的数据库备份。若缺少数据库环境变量或本机没有 `mysqldump`，脚本会在 `MANUAL_ACTIONS.txt` 中记录原因。
- `uploads.zip`：上传文件归档。若上传目录不存在，脚本会在 `MANUAL_ACTIONS.txt` 中记录原因。

## 恢复演练

恢复演练必须在隔离环境执行，不得直接覆盖生产库。

1. 新建空 MySQL 数据库，并准备与生产同版本的 Java 后端配置。
2. 导入备份：

```powershell
mysql --host <host> --port <port> --user <user> --password < database.sql
```

3. 解压 `uploads.zip` 到隔离环境的 `server/uploads`。
4. 启动 Java 后端和管理端：

```powershell
npm run start:all
```

5. 打开管理端系统设置，检查“日志管理”和“保留策略”；再按 `docs/小程序关键路径冒烟测试清单.md` 至少验证 A-01、F-05、F-08、S-03、S-04。
6. 记录恢复耗时、缺失文件、失败接口和修复动作。

## 频率和责任

- 封闭试点期间每日执行一次备份。
- 每周至少执行一次隔离恢复演练。
- 管理员负责确认 `manifest.json` 没有未处理的 `manual_actions`。
- 试点结束或家庭退出时，应先完成一次备份，再处理隐私删除或更正请求。
