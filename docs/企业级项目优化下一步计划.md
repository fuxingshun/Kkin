# KinEcho 企业级扩展与优化下一步计划

## Summary

基于 2026-05-08 对当前项目的审计，KinEcho 已具备封闭试点基础：微信小程序三端、React 管理后台、Spring Boot/MySQL 后端、CI 门禁、会话鉴权、隐私请求、备份脚本和试点文档均已成型。

当前验证结果：

- Java 后端 156 项测试通过。
- 后台管理端构建通过。
- 小程序构建通过，65 项冒烟预检通过。
- 根项目 `npm audit` 未发现漏洞。
- 小程序 high/critical 依赖审计已通过；剩余 `miniprogram-simulate -> postcss@7.0.39` moderate 漏洞按 `docs/依赖审计豁免清单.md` 跟踪。

下一步目标不是继续堆功能，而是把“试点可用”升级为“企业可运营、可审计、可扩展、可恢复”的生产级系统。

## Key Changes

### P0：生产安全止血，1 周

- 将生产默认改为安全优先：强制 `KINECHO_API_TOKEN_ENABLED=true`、`KINECHO_FAMILY_SCOPE_SESSION_REQUIRED=true`、`KINECHO_PHONE_SUFFIX_LOGIN_ENABLED=false`。
- 禁止默认 `admin/service` 口令、开发签名密钥、演示登录策略进入生产环境。
- 配置 CORS 白名单，避免长期使用全开放跨域策略。
- 重新评估 `/psychology-videos/{id}.mp4` 与 `/uploads/ai-voice/**` 的公开访问策略。
- 上传文件增加 MIME、文件头魔数、大小分层、危险扩展名和内容伪装拦截。
- 处理小程序依赖审计风险，优先定位并修复或隔离 `vm2`、`axios`、`html-minifier` 等高危来源。

### P1：账号权限与租户隔离，2 周

- 用正式 IAM/RBAC 替换静态配置账号。
- 管理员、服务人员、咨询师、家属、老人均使用数据库账号、密码哈希、角色权限、禁用/锁定状态和操作审计。
- 所有家庭数据接口从服务端 session 推导 `family_id`。
- 前端传入的 `family_id` 仅作为管理端授权筛选条件，不作为普通用户访问依据。
- 补齐服务人员机构、家庭授权模型，避免 `service-family-id=family_001` 成为长期限制。
- 增加登录失败锁定、session 失效、令牌轮换和管理员操作二次确认机制。

当前落地进展：

- `auth_accounts` / `auth_audit_logs` 已覆盖数据库账号、PBKDF2 密码哈希、权限、禁用、失败计数、锁定时间与登录审计。
- 管理员、服务人员登录已优先使用 `auth_accounts`，静态账号仅作为显式开启的兼容回退；生产启动校验会阻止静态账号进入 `prod/production`。
- 家属、老人登录已支持通过 `auth_accounts.user_id` 绑定到 `users` 记录并使用哈希密码、锁定和审计；未迁入账号时保留本地/试点兼容登录路径。
- 管理端账号生命周期接口已补齐：`GET /api/admin/auth/accounts`、`POST /api/admin/auth/accounts`、`PUT /api/admin/auth/accounts/{accountId}` 支持账号查询、创建、禁用/启用、解锁、权限调整和密码轮换；接口要求 admin session，密码、禁用和解锁类敏感操作要求显式二次确认字段。
- `/api/admin/**` 已统一纳入 admin session 拦截，缺失 session 返回 401，非 admin session 返回 403，避免管理端接口只依赖通用 API Token。
- 数据库账号 session 已加入 `auth_account_id` 与 `session_version`，敏感账号变更会提升版本；`/api/admin/**` 会拒绝已轮换或已禁用 admin 账号的旧 session。
- `/api/admin/auth/**` 已开始按权限粒度收敛，要求 `admin:auth`、`admin:*` 或本地兼容 `admin:legacy` 权限；账号生命周期服务层也执行同样权限校验。
- `GET /api/me` 已返回角色、权限、机构范围、家庭授权范围和 session 过期时间。

### P2：后端架构拆分与数据治理，2-3 周

- 拆分 `KinEchoApiService` 和 `KinEchoMapper`，按 auth、family/user、alert/task、media、mood/mental、consultation、admin、ai 八个领域拆分 service、controller、repository。
- 引入 DTO、参数校验、统一错误码和 OpenAPI 文档。
- 减少 `Map<String, Object>` 在公开接口中的使用。
- 用 Flyway 或 Liquibase 替换启动时建表和迁移。
- 演示 seed、测试数据、生产 schema 分离管理。
- 迁移失败必须显式失败并告警，不再吞异常。

当前落地进展：

- 服务层错误响应已开始统一为 `{ code, message, request_id }`，同时保留旧版 `error` 字段，避免一次性破坏现有前端错误处理。
- API Token、admin session、family scope 等拦截器拒绝响应已改为同一 JSON 错误结构，避免 `sendError` 返回容器默认错误页。
- 已引入 `spring-boot-starter-validation`，并将管理端账号创建/更新接口迁移到 `AdminAuthAccountCreateRequest` / `AdminAuthAccountUpdateRequest` DTO 与 Jakarta Validation；validation 失败会返回 `{ code, message, request_id, details }` 结构化错误。
- 隐私请求创建与后台审核接口已迁移到 `PrivacyRequestCreateRequest` / `PrivacyRequestReviewRequest` DTO，保留 `family_id`、`request_type`、`process_note` 等现有 JSON 字段，同时前置校验请求类型、审核状态和文本长度。
- 已接入 `springdoc-openapi-starter-webmvc-ui`，提供 `/v3/api-docs` 与 `/swagger-ui.html` 的 OpenAPI 骨架，并按 `admin-api`、`service-api`、`miniapp-api` 分组；生产示例默认关闭，可通过 `KINECHO_OPENAPI_ENABLED` / `KINECHO_SWAGGER_UI_ENABLED` 显式开启。
- 管理端账号与隐私请求 DTO 已补充 `@Schema` 描述、示例和枚举值，使 OpenAPI 输出具备可读请求体契约，便于后续前端类型生成。
- 服务人员认证提交与后台审核接口已迁移到 `ServiceCertificationSubmitRequest` / `ServiceCertificationReviewRequest` DTO，前置校验身份字段、审核状态和拒绝原因长度，并补充 OpenAPI schema 元数据。
- 用户创建与更新接口已迁移到 `UserCreateRequest` / `UserUpdateRequest` DTO，保留 `user_type`、`family_id`、`wechat_openid`、`operator` 等现有字段，同时前置校验家庭范围、用户类型和文本长度。
- 服务随访创建、随访状态推进和服务记录创建接口已迁移到 `ServiceFollowupCreateRequest` / `ServiceFollowupStatusRequest` / `ServiceRecordCreateRequest` DTO，覆盖 `consultation_type`、`scheduled_time`、`cancel_reason`、`alert_id` 等照护闭环字段。
- 咨询预约创建与更新接口已迁移到 `ConsultationCreateRequest` / `ConsultationUpdateRequest` DTO，覆盖 `notify_service`、`concern_level`、`cancel_reason`、`counselor_id` 等预约与服务协同字段。
- 家庭日程创建/更新与老人端日程状态推进接口已迁移到 `ScheduleCreateRequest` / `ScheduleUpdateRequest` / `ScheduleStatusRequest` DTO，覆盖 `schedule_type`、`schedule_time`、`repeat_type`、`auto_remind` 与 `status` 等照护提醒字段。
- 家庭告警创建/处理/回复和家庭留言创建接口已迁移到 `FamilyAlertCreateRequest` / `FamilyAlertHandleRequest` / `FamilyAlertReplyRequest` / `FamilyMessageCreateRequest` DTO，覆盖 `alert_type`、`level`、`metadata`、`reply_message`、`sender_relation` 与 `scheduled_time` 等家庭通信字段。
- 隐私同意记录和家庭绑定码接口已迁移到 `ConsentRecordCreateRequest` / `FamilyBindByCodeRequest` DTO，覆盖 `consent_type`、`accepted`、`actor_role`、`metadata`、`binding_code` 与 `wechat_openid` 等合规和绑定字段。
- 老人端告警创建和心情记录接口已迁移到 `ElderlyAlertCreateRequest` / `ElderlyMoodCreateRequest` DTO，覆盖 `alert_type`、`level`、`metadata`、`mood_type`、`mood_score`、`trigger_event` 与 `recorded_at` 等照护采集字段。
- 家庭媒体更新、老人端媒体播放记录和媒体反馈接口已迁移到 `MediaUpdateRequest` / `MediaPlayRecordRequest` / `MediaFeedbackRequest` DTO，覆盖 `tags`、`time_windows`、`cooldown`、`priority`、`duration_watched`、`completed` 与 `feedback_type` 等媒体互动字段。

### P3：合规、可观测与运维，2 周

- 建立数据分类和保留任务：聊天、录音、AI 音频、心理筛查帧、咨询记录、审计日志分别配置自动清理、导出、删除和留痕策略。
- 增加结构化日志、指标、链路追踪、错误告警和健康检查分级。
- 建立备份调度、恢复演练记录、RPO/RTO 和故障复盘模板。
- 提供 Docker/Compose 或等价部署模板。
- 补齐环境变量校验、启动前配置检查、生产回滚说明和版本发布记录。

当前落地进展：

- `/api/health` 已从平面 `ok` 响应升级为分级组件健康检查，覆盖数据库、上传目录、AI 音频目录、AI 语音上传目录和关键安全配置；保留 `status/timestamp/backend` 兼容字段。
- 新增 `/api/admin/ops/metrics`，输出高风险队列、未读提醒、待处理隐私请求、待审核服务认证、活跃账号和锁定账号等运营指标；单项采集失败会标记为 `critical` 并将整体状态降级。

### P4：前端体验与运营闭环，2 周

- 管理后台从 `src/admin/main.tsx` 单文件拆成页面和业务组件。
- 补齐权限菜单、审计查询、服务 SLA、风险升级、咨询师排班与内容审核。
- 小程序 `app.css` 按老人、家属、服务端拆分样式，降低维护成本。
- 老人端首屏继续压缩为陪伴、联系家人、今日提醒、心情记录。
- 重新开启或替代小程序生产压缩策略，在包体、调试能力和安全之间做明确取舍。

当前落地进展：

- 管理后台系统设置页已接入 `/api/admin/ops/metrics`，可查看风险队列、隐私请求、认证审核、活跃账号和锁定账号等运维指标，并支持手动刷新；冒烟清单新增后台运维指标闭环校验。

## Public APIs / Interfaces

- 新增或标准化 `/api/auth/*`、`/api/admin/*`、`/api/service/*` 的权限模型。
- 登录态接口返回明确的角色、权限列表、机构范围和家庭授权范围。
- 家庭域接口逐步移除普通用户必传 `family_id`，改为服务端从 session 注入。
- 管理端保留显式 `family_id`，但必须经过授权校验。
- 输出 OpenAPI 文档并作为前端类型生成来源。
- 统一错误结构为 `{ code, message, request_id }`。
- 数据库迁移进入版本化目录，seed 数据、测试数据、生产数据初始化分开管理。

## Test Plan

- 保留现有门禁：
  - `npm run build:admin`
  - `npm --prefix miniapp run build:weapp`
  - `mvn test -f server-java/pom.xml`
  - `npm --prefix miniapp run smoke:checklist`
- 新增 MySQL 集成测试、API contract 测试、跨家庭越权测试、上传安全测试、隐私导出/删除测试、AI 降级与危机升级测试。
- 将 `smoke:checklist -- --require-api` 和受控开发库 `--live-user-crud` 纳入预发布流水线。
- 增加依赖审计门禁：根项目和小程序均不能存在 critical/high 漏洞。
- 确需豁免的依赖漏洞必须写明来源、影响面、临时缓解方式和到期时间。
- 补齐真机测试：录音、摄像头、弱网、低性能设备、大字体、高对比、登录态过期和服务端不可用场景。

## Assumptions

- 产品继续按“封闭试点 -> 可控扩容 -> 企业部署”推进，不直接公众开放。
- KinEcho 定位为陪伴、照护协同和风险筛查参考，不提供医学诊断或急救替代。
- 下一步优先级以安全、权限、数据、运维、质量为先，新增业务功能让位于企业级治理。
- 文档中的周期为建议节奏，实际执行可按团队人力和试点上线压力调整。
