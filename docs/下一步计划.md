# KinEcho 下一步计划

分析日期：2026-05-08  
分析范围：根项目、`miniapp/` 微信小程序、`server-java/` Java 后端、`src/admin/` 管理端、`docs/` 文档和 `scripts/` 工程脚本。

## 1. 当前项目画像

KinEcho 当前已经不是单一原型项目，而是一个三端业务闭环仓库：

- 微信小程序：`miniapp/`，包含老人端、家属端、服务人员端和登录/法务页面，共 47 个页面入口。
- Java 后端：`server-java/`，Spring Boot 3、Java 21、MySQL、MyBatis-Plus，承载鉴权、家庭数据、媒体、咨询、服务工单、AI 陪伴、隐私请求和管理端接口。
- 管理端：`src/admin/`，React + Vite 单页后台，已覆盖总览、用户、内容、预警、服务、数据分析、设置等管理能力。
- 工程化：已有 GitHub Actions CI、根项目和小程序依赖审计、后台构建、小程序构建、Java 测试、小程序冒烟脚本和备份脚本。
- 运营文档：已有封闭试点上线、安全验证、备份恢复、每日运营、试点名单、依赖审计豁免等文档。

总体判断：项目已经具备封闭试点和内部演示基础，下一步重点不应继续堆页面，而应优先把系统从“能跑通”提升到“可长期维护、可审计、可扩容、可上线交付”。

## 2. 关键发现

### 2.1 后端能力完整，但领域边界过重

当前 `KinEchoApiService.java` 约 5210 行，`KinEchoMapper.java` 约 1398 行，`KinEchoApiController.java` 约 697 行。后端已经覆盖大量业务能力，但多数领域仍集中在一个 service、一个 mapper、一个 controller 中。

主要问题：

- `KinEchoApiService` 同时处理登录、微信绑定、家庭消息、情绪、日程、媒体、心理筛查、咨询、服务工单、管理端统计、AI 陪伴和运维健康检查，修改任何业务都容易影响其它业务。
- `KinEchoMapper` 同时承担建表、补字段、索引、seed 数据、通用 SQL 工具和部分迁移逻辑，启动期数据库治理风险偏高。
- 公开接口仍大量使用 `Map<String, Object>` 作为入参和出参，虽然已经开始补 DTO，但类型契约还没有完全稳定。
- 建表和迁移依赖运行时代码执行，缺少 Flyway/Liquibase 这类版本化迁移目录，生产环境回滚、审计和灰度发布会比较吃力。

建议方向：

- 按领域拆出 `auth`、`family`、`elderly`、`media`、`consultation`、`service`、`admin`、`ai`、`ops` 模块。
- 每个领域形成 `Controller -> Service -> Repository/Mapper -> DTO` 的独立边界。
- 优先迁移写接口 DTO 和统一响应模型，再逐步收敛查询接口返回类型。
- 将 `CREATE TABLE`、`ALTER TABLE`、seed 数据从 Java 代码中迁出，建立版本化迁移和环境隔离 seed。

### 2.2 安全和租户隔离已有进展，但默认家庭 ID 仍需收敛

项目已经有 API Token、session token、admin session、family scope 拦截、生产启动安全校验、受控媒体下载、AI 音频短期签名 URL、隐私请求和审计日志，这些是很好的基础。

主要问题：

- 小程序仍保留 `DEFAULT_FAMILY_ID = 'family_001'`，多个服务层 API 默认拼接或回退到 `family_001`。
- 管理端仍有 `ADMIN_FAMILY_ID = import.meta.env.VITE_ADMIN_FAMILY_ID || 'family_001'`，虽然已有家庭选择器，但长期依赖默认家庭会限制多家庭、多机构扩展。
- 服务人员端仍需要更正式的机构、家庭授权、值班范围和可见数据范围模型。
- admin 权限粒度已有雏形，但前端菜单、按钮级权限、审计查询和敏感操作复核还需要系统化。

建议方向：

- 普通老人端、家属端、服务端接口全面改为从 session 推导家庭上下文，前端只在管理端或明确授权筛选时传 `family_id`。
- `family_001` 只能作为本地演示 seed，不应出现在生产默认路径和核心 SDK 默认值中。
- 补齐组织、服务人员、家庭授权关系表，替代单个 `service-family-id`。
- 管理端根据 `GET /api/me` 返回的权限动态生成菜单、按钮和操作范围。
- 增加登录限流、Token 轮换、session 续期/撤销、管理员敏感操作二次确认审计。

### 2.3 小程序功能丰富，但样式和服务层需要拆分

小程序页面覆盖老人、家属、服务三端，业务链路较完整。当前 `miniapp/src/app.css` 约 13724 行，`family.ts` 约 723 行，`elderly.ts` 约 693 行，`request.ts` 约 426 行。

主要问题：

- 全局样式过大，三端样式混在 `app.css` 中，后续维护和回归成本高。
- 老人端页面数量较多，首版应继续收敛为陪伴、联系家人、今日提醒、心情记录、回忆播放几个强路径。
- 服务层类型定义、请求函数、数据转换混在同一文件中，后续多人协作容易冲突。
- 生产构建中小程序压缩被关闭，虽然有包体预算脚本，但长期会影响包体和首屏性能。
- 真机测试、弱网、权限拒绝、低性能设备、大字体和高对比仍主要依赖人工清单。

建议方向：

- 将 `app.css` 拆成 `styles/common.css`、`styles/elderly.css`、`styles/family.css`、`styles/service.css`，再按页面逐步下沉。
- 将 `services/elderly.ts`、`services/family.ts` 拆成领域文件，例如 `schedule`、`message`、`media`、`mood`、`consultation`。
- 建立小程序端通用数据状态和错误状态规范，减少每个页面重复写 loading/error/empty 逻辑。
- 重新评估 Taro 生产压缩禁用原因，形成“可压缩路径”或明确的替代方案。
- 对老人端核心页面做真机适老化验收，重点检查字号、触控目标、语音权限、弱网重试和离线提示。

### 2.4 管理端已经可用，但需要模块化和权限化

管理端 `src/admin/main.tsx` 约 2151 行，`src/admin/api.ts` 约 691 行，当前所有页面、弹窗、转换函数、状态逻辑基本集中在少数文件中。

主要问题：

- 页面和业务组件未拆分，后续添加审计、账号、内容、排班、SLA 等能力会让主文件继续膨胀。
- 表格、筛选、弹窗、表单、状态徽标等管理后台通用控件没有沉淀。
- API 类型手写维护，后端已有 OpenAPI 骨架但前端尚未自动生成类型。
- 管理端登录态存储在 localStorage，后续需要配合 session 过期、刷新和强制下线策略。
- 运营指标已有入口，但缺少告警规则配置、历史趋势和处置闭环。

建议方向：

- 拆分为 `pages/`、`components/`、`hooks/`、`api/`、`types/`。
- 引入统一的 `DataTable`、`PageHeader`、`ConfirmDialog`、`FormField`、`StatusBadge`。
- 用 OpenAPI 生成前端类型，减少手写接口漂移。
- 按权限渲染导航和操作按钮，禁止无权限用户看到或触发敏感动作。
- 扩展运维页：健康检查历史、依赖审计状态、备份状态、隐私请求 SLA、风险工单 SLA。

### 2.5 AI 和心理关怀需要“安全边界 + 可评估性”

项目已经接入百炼 ASR、对话、TTS，并有 AI 危机关键词识别和家属/服务提醒。文档中仍出现模型名不一致的迹象，例如 README 中写 `qwen3.5-plus`，配置中默认 `qwen-turbo`。

主要问题：

- AI 模型、提示词、超时、降级策略和安全规则还没有形成版本化配置和评估报告。
- 危机识别主要依赖关键词，容易漏报或误报，需要运营可配置规则和人工复核。
- 心理筛查、心理视频、咨询服务之间需要更明确的“非医学诊断”边界和转人工机制。
- AI 互动数据保留策略已有配置项，但需要确认后台清理任务、导出和删除是否真正闭环。

建议方向：

- 建立 AI 配置版本表：模型、系统提示词、安全提示词、温度、超时、发布时间、负责人。
- 建立 AI 回归评估集：陪伴类、求助类、危机类、医疗边界类、儿童/老人隐私类。
- 危机识别升级为“规则 + 模型判断 + 人工处置状态”的闭环。
- 管理端增加 AI 失败率、平均响应时间、危机触发数、人工跟进结果。
- 统一文档和配置中的模型名，避免上线时对供应商能力产生误解。

### 2.6 数据治理和合规已经起步，但还缺后台任务

项目已有 `consent_records`、`privacy_requests`、`care_audit_logs`、保留周期配置和数据导出/删除请求入口。

主要问题：

- 数据保留周期目前更像配置和展示，仍需确认定时清理、清理审计和异常告警。
- 删除/更正请求的实际数据处理范围需要明确定义，尤其是聊天、录音、AI 音频、上传媒体、咨询记录、审计日志。
- 数据导出需要形成稳定包格式、字段脱敏策略和下载有效期。
- 文档中部分数据库说明已经落后于当前实现，应同步更新。

建议方向：

- 增加后台定时任务：AI 音频清理、语音上传清理、心理筛查帧清理、过期会话清理。
- 每次清理写入 `care_audit_logs` 或独立 `data_retention_jobs` 表。
- 定义隐私请求处理 SOP：受理、确认身份、执行、复核、通知、归档。
- 将数据导出做成异步任务，返回短期签名下载地址。
- 更新 `docs/数据库表结构说明.md`，补齐当前所有表和字段来源。

### 2.7 工程化基础不错，但还需要更硬的质量门禁

当前 CI 已覆盖依赖审计、后台构建、小程序构建、Java 测试、小程序冒烟预检。小程序还使用本地 vendor 包隔离部分构建链依赖风险。

主要问题：

- 本次分析未重新执行完整构建和测试，只做静态审计和文件结构检查。
- Java 测试数量已经不少，但主要集中在规则和 DTO，缺少真实 MySQL 集成测试、跨家庭越权集成测试、上传下载端到端测试。
- 小程序冒烟脚本很完整，但它通过读源码断言较多，容易和实现细节强绑定。
- 缺少管理端自动化 E2E 测试。
- 小程序依赖 override 和本地 vendor 包需要长期维护，到期复核不能只靠人工记忆。

建议方向：

- 保留当前 CI，并增加分层测试：Java 单元、Java 集成、API contract、管理端 E2E、小程序核心路径真机/模拟检查。
- 增加 `npm run typecheck:miniapp`、`npm run lint` 或等价静态检查。
- 引入预发布环境门禁：`smoke:checklist -- --require-api` 和受控开发库 `--live-user-crud`。
- 将依赖豁免到期日期纳入 CI 或脚本检查。
- 为备份恢复脚本增加 dry-run 和恢复校验脚本。

### 2.8 文档数量充足，但需要统一口径

`docs/` 中已有多份试点、安全、备份、清理和计划文档，覆盖面较好。

主要问题：

- `README.md`、`server-java/README.md`、`application.yml` 中的 AI 模型名和生产配置描述需要统一。
- `docs/数据库表结构说明.md` 对当前表结构、配置方式和新增合规表描述不完整。
- 已有 `企业级项目优化下一步计划.md` 更偏企业治理，本文件建议作为项目总路线图，后续需要避免两份计划互相发散。
- 部分文档是“准备清单”，缺少实际执行记录链接和负责人字段。

建议方向：

- 建立 `docs/README.md` 或文档索引，标明每份文档的适用场景和维护责任人。
- 将过期内容标记为“历史”，避免新成员误用旧配置。
- 每次发布后更新“发布记录 + 验证记录 + 已知风险”。
- 将真实试点执行记录单独放在 `docs/pilot-records/` 或受控私有位置。

## 3. 优先级路线图

### P0：一周内完成的稳定性收敛

目标：减少上线前最容易造成事故的配置、文档和默认值问题。

- 统一 README、后端 README、配置文件中的 AI 模型和生产配置说明。
- 建立默认家庭 ID 清单，标记哪些是演示 fallback，哪些必须移除。
- 更新数据库表结构文档，补齐当前合规、账号、服务、心理内容和媒体相关表。
- 为 `docs/企业级项目优化下一步计划.md` 和本文件建立关系说明，避免计划重复维护。
- 执行一次完整门禁并记录结果：
  - `npm run audit:security`
  - `npm run build:admin`
  - `npm --prefix miniapp run build:weapp`
  - `mvn test -f server-java/pom.xml`
  - `npm --prefix miniapp run smoke:checklist`
- 确认小程序生产压缩关闭的根因，并输出可恢复压缩的技术方案或明确的风险接受记录。

### P1：两周内完成后端领域拆分第一阶段

目标：先拆最容易变化和最有风险的领域。

- 拆出 `auth`：登录、session、账号、权限、审计。
- 拆出 `family`：家庭用户、绑定码、家庭范围校验、家属/老人基础资料。
- 拆出 `media`：上传、下载、缩略图、播放历史、反馈、文件安全。
- 拆出 `service`：工单、个案、随访、SLA、服务记录。
- 为上述领域建立 DTO、响应类型和 mapper/repository。
- 保留旧接口路径不变，内部逐步迁移，避免一次性破坏小程序和管理端。

### P2：两到三周内完成数据迁移和租户治理

目标：让数据库和多家庭权限进入生产可治理状态。

- 引入 Flyway 或 Liquibase，迁移当前建表和补字段逻辑。
- 明确 seed 策略：本地演示 seed、测试 seed、生产初始化分离。
- 建立组织、服务人员、家庭授权模型。
- 普通端接口逐步停止信任前端 `family_id`。
- 管理端保留家庭筛选，但必须由权限范围决定可选家庭列表。
- 增加跨家庭越权集成测试和管理端授权测试。

### P3：两周内完成前端模块化和核心体验优化

目标：降低前端维护成本，并提升老人端真实可用性。

- 管理端拆分 `src/admin/main.tsx`，先拆页面，再拆通用控件。
- 管理端按权限渲染导航和按钮。
- 小程序拆分 `app.css`，建立三端样式边界。
- 小程序拆分 `services/elderly.ts`、`services/family.ts`。
- 老人端首页继续压缩为四个核心动作：陪我聊、联系家人、今日提醒、记录心情。
- 对老人端进行真机适老化测试，记录大字体、高对比、弱网、权限拒绝结果。

### P4：两周内完成运维、合规和质量门禁

目标：让系统可以被运营和维护，而不是只由开发人员临时排查。

- 增加结构化日志、请求 ID 贯穿、关键业务审计查询。
- 增加后台保留任务和清理任务执行记录。
- 管理端扩展运维指标趋势、告警阈值和处理记录。
- 备份脚本增加恢复校验流程。
- 增加 API contract 测试、真实 MySQL 集成测试、上传安全测试。
- 小程序和管理端增加关键路径自动化测试。

### P5：试点后扩展方向

目标：在封闭试点证明可用后，支持可控扩容。

- 多机构管理：机构、服务队伍、咨询师、家庭授权和数据隔离。
- 服务 SLA：高风险响应时限、超时升级、值班排班、处置复盘。
- 内容运营：心理视频、问答、文章、审核状态、投放范围和版本管理。
- AI 运营：模型配置、提示词版本、危机样本评估、失败率和人工复核。
- 商业化准备：企业部署包、环境检查脚本、版本发布说明、回滚方案。

## 5. 验收指标

### 技术指标

- `KinEchoApiService.java` 降到 1500 行以内，且没有跨领域业务新增继续写入该类。
- `KinEchoMapper.java` 不再负责生产建表和迁移。
- 公开写接口 90% 以上使用 DTO 和 validation。
- 普通端接口不再依赖前端传入 `family_id` 决定数据范围。
- 管理端主入口文件低于 500 行，页面和通用组件清晰拆分。
- 小程序全局 `app.css` 降到 3000 行以内，其余样式按端或页面归档。

### 质量指标

- CI 全绿：依赖审计、后台构建、小程序构建、Java 测试、小程序冒烟。
- 关键 API 有 contract 测试。
- 跨家庭访问、未登录访问、错误 Token、上传伪装文件均有自动化或半自动化验证。
- 小程序真机测试覆盖至少 1 台 iOS、1 台 Android、1 台低性能设备。
- 备份恢复演练能在测试环境完成并记录 RPO/RTO。

### 业务指标

- 高风险工单有 SLA、负责人、处理记录和超时升级。
- AI 失败率、危机触发率、人工跟进完成率可在管理端查看。
- 隐私导出、删除、更正请求有状态流转和审计记录。
- 试点家庭、服务人员、咨询师、管理员名单冻结并可追溯。

## 6. 当前不建议做的事

- 不建议继续新增大量页面，除非先完成服务层和样式拆分。
- 不建议在 `KinEchoApiService` 中继续叠加新业务。
- 不建议把 `family_001` 作为真实环境默认逻辑继续扩散。
- 不建议跳过真实 MySQL 迁移治理直接进入多机构扩容。
- 不建议将 AI 能力包装成医学诊断或急救替代能力。
- 不建议长期依赖手工清单代替关键安全和权限自动化测试。

## 7. 推荐执行顺序

1. 先做 P0，修正文档、默认值、验证记录和压缩策略。
2. 然后做后端拆分和数据库迁移，因为它决定后续所有功能的维护成本。
3. 同步做管理端和小程序模块化，避免新功能继续堆在大文件里。
4. 再补自动化测试、备份恢复、合规清理和运维观测。
5. 最后再进入多机构、SLA、AI 运营和商业化部署扩展。
