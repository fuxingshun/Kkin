# KinEcho 封闭试点上线方案

## Summary

目标按“封闭试点 + 真人咨询闭环 + 家属和服务人员共同跟进”推进，不做公众开放，不把产品表述为医学诊断工具。首版价值应聚焦：老人获得低门槛陪伴和求助，家属看到可信的状态摘要，服务人员能处理高风险或持续异常。

当前项目已有老人端、家属端、服务端、管理端、Java 后端和 MySQL 数据结构雏形。验证结果：后台构建通过，小程序构建通过但有包体警告；Java 测试失败；小程序冒烟脚本失败，原因是脚本仍依赖已不存在的 `miniapp/src/constants/psychologyVideos.ts`。此外 `package.json` 的 `start:all/stop:all` 指向不存在的 `scripts/` 目录。

## 当前不足

- 账号和权限还不适合真实上线：目前主要靠静态 API Token、手机号后 6 位/默认密码、前端传 `family_id`；缺少按登录用户推导家庭权限的服务端鉴权，管理端也默认固定 `family_001`。
- 心理健康能力边界不够严谨：当前“心理关怀检测”主要依据摄像头动作、活体和画面质量生成风险提示，不是可靠心理评估；AI 陪伴也缺少自伤/危机识别、转人工策略和免责声明分层。
- 真人咨询闭环不足：已有咨询师和咨询记录表，但缺少咨询师资质审核、排班、咨询记录权限、服务质量追踪、取消/改约规则、家属可见范围和管理端审核能力。
- 家属与服务人员闭环还偏弱：服务端能看工单和个案，但缺少 SLA、超时提醒、值班分配、处理备注模板、风险升级和事后回访机制。
- 隐私与合规不足：登录协议只是弹窗文案；摄像头帧、语音、上传媒体通过本地 `/uploads/**` 暴露，缺少访问控制、保留周期、删除/导出、同意记录、操作审计和备份策略。
- 内容运营未稳定：心理视频/问答依赖数据库，但仓库没有正式种子或后台维护闭环；冒烟脚本与当前实现脱节。
- 老人端 UX 需要收敛：已有大字体和高对比偏好，但页面多、文字密、功能入口多；首版应把老人端压缩成“陪我聊、联系家人、今日任务/心情”几个稳定动作。
- 发布工程不足：没有 CI 门禁；`mvn test` 当前失败；冒烟脚本失败；生产配置、监控、日志、备份、告警和回滚方案未成体系。

## Key Changes

- 账号权限：新增服务端会话/JWT 或等价 session，所有接口从会话推导 `role/user_id/family_id`；禁止普通端用请求参数越权访问其他家庭。管理端增加家庭/机构选择和角色权限。
- 心理支持：保留“非诊断”定位，新增量表型自评入口和风险分级规则；AI 陪伴加入危机关键词识别、转家属/服务工单、紧急提示和对话安全边界。
- 真人咨询：补齐咨询师资质、可预约时段、预约生命周期、服务记录、家属可见摘要、取消/改约、咨询后回访；服务人员端处理高风险工单并写入记录。
- 隐私合规：新增正式用户协议/隐私政策页面、同意记录表、上传文件鉴权、音频/图片保留周期、用户数据导出/删除、审计日志和备份恢复。
- 运营后台：管理端拆分并补齐服务认证审核、咨询师管理、心理内容管理、风险工单看板、导出和健康检查；修复 `ADMIN_FAMILY_ID` 单家庭默认限制。
- 发布质量：修复 Java 测试和小程序冒烟脚本，补齐缺失启动脚本或删除无效命令，建立 CI 门禁：`npm run build:admin`、`npm --prefix miniapp run build:weapp`、`mvn test`、`npm --prefix miniapp run smoke:checklist` 必须通过。

## Implementation Plan

1. 第 0 阶段，1 周：上线前止血  
   修复失败测试和冒烟脚本，清理失效脚本；确认生产配置只走环境变量；建立 CI；冻结首批试点家庭、服务人员、咨询师名单；明确“非医学诊断”的产品文案。

2. 第 1 阶段，1-2 周：真实账号与数据隔离  
   完成微信生产登录、会话鉴权、角色权限、家庭绑定码、管理端家庭/机构选择；所有家庭数据读写改为服务端鉴权范围内访问；补服务认证审核后台。

3. 第 2 阶段，2 周：心理支持与真人咨询闭环  
   增加老人端轻量自评、AI 风险识别、家属提醒、服务工单、咨询预约、咨询记录和回访；高风险路径固定为“老人触发/系统识别 -> 家属通知 -> 服务人员处理 -> 家属看到处理结果”。

4. 第 3 阶段，1-2 周：适老化体验重构  
   老人首页首屏只保留核心动作；统一 48px 以上触控目标、可读字体、语音播报、慢网络提示、失败重试；家属端突出“今天是否需要我做什么”；服务端突出待处理优先级。

5. 第 4 阶段，1-2 周：隐私、运维和内容  
   上传文件改为受控访问；建立音频、截图、聊天、咨询记录保留策略；补数据备份和恢复演练；心理视频/问答进入后台维护；增加服务监控、错误日志和告警。

6. 第 5 阶段，2 周：封闭试点  
   先给 10-30 户真实家庭使用，配 1-2 名服务人员和固定咨询师；每日检查高风险工单、AI 失败率、老人使用率、家属查看率和咨询完成率；试点结束后再决定是否扩大。

## Public Interfaces / Data

- 新增 `GET /api/me` 或等价会话接口，返回当前用户、角色、家庭、老人绑定状态。
- 所有家庭接口移除对前端 `family_id` 的信任，保留管理端受权限控制的家庭选择。
- 新增 consent/audit 数据：记录协议版本、心理筛查同意、咨询同意、数据导出/删除操作。
- 扩展咨询相关数据：咨询师资质、可预约时段、预约状态、服务记录、回访记录、家属可见摘要。
- 上传资源由公开 `/uploads/**` 改为鉴权下载或短期签名 URL。

## Test Plan

- 单元测试：鉴权、家庭隔离、绑定码、服务认证审核、风险分级、咨询状态机、文件访问权限。
- 集成测试：老人建档、家属绑定、AI 陪伴、心情记录、心理自评、高风险工单、服务处理、家属报告。
- 小程序真机测试：老人端大字体/高对比/语音/摄像头/录音/弱网/无权限场景。
- 安全测试：跨家庭访问、未登录访问、上传文件类型、静态资源越权、Token 泄露、默认密码禁用。
- 发布门禁：四条构建/测试命令全绿，且人工冒烟清单覆盖老人、家属、服务、管理端关键路径。

## Assumptions

- 首版只做封闭试点，不做公众上线。
- 产品定位是真人咨询闭环下的心理关怀支持，不做医学诊断。
- 高风险由家属和服务人员共同跟进，服务人员需要后台审核和明确 SLA。
- 真实上线前必须先修复当前 Java 测试、冒烟脚本和鉴权隔离问题。
